Kendo UI - Opinionated Guide For Building Mobile Apps

DRAFTby cody lindley

Introduction

What is this?

This is an opinionated guide for building websites, applications, and hybrid applications running on phones and tablets using Kendo UI mobile widgets and application tools. The intent is for the prescriptions found within this document to become a style used by developers when implementing Kendo UI mobile solutions in a browser or webview.

If you are looking for information about using Kendo UI with Angular to build a mobile application please refer to the following documentation, "Building applications with Kendo UI Mobile and AngularJS".

Why use Kendo UI to build mobile apps?

Kendo UI provides specific touch first widgets to be used in the construction of phone and tablet web based interfaces. In addition to widgets, Kendo UI also provides application logic for building a Single Page Application in a mobile browser or webview (i.e. Application, View, Layout). In this guide I will refer to the application logic that is provide by Kendo UI as "mobile application logic".

While Kendo UI widgets can run stand alone without an active mobile application instance this guide assumes the use of the mobile application logic in conjunction with the mobile widgets. Unless you are bringing a replacement for the application logic provided by kendo.mobile.Application (e.g. Angular + ng-view + ng-router) the application logic will almost always be needed.

Who should read this

You are front-end engineer who has experience developing applications in HTML, CSS, and JavaScript (e.g. SPA applications using MVVM) that wants to use these skills to build a mobile/tablet web site/application or a hybrid native application and are in need of some direction/opinions to get you started.

1. Kendo Mobile Do's

1.1 Take stock of technologies available on the web platform

A Kendo UI mobile app, even if it's running in a webview, is render by a browser engine (i.e. a webview is like a browser window tab). Make sure to take note of the HTML5 (aka JavaScript API's), CSS, and JavaScript standards that are support by the browser engine(s) that will be running your application.

It's likely the case that the browser engine you use to render a Kendo UI mobile application will be a newer, modern browser engine, that supports things like ES5, Flexbox, CORS, and Web Sockets.

Make sure you are aware of everything you have at your disposal for development and use it.

Notes
  • Kendo UI mobile is supported on the following operating systems and browser combinations Android 2.3+, iOS 6+, Windows Phone 8+, Chrome for Android. Kendo UI mobile widgets used without the mobile application logic will run on OS X 10.5+ (IE10+, Chrome Recent, Opera 15+, Safari 5+) and Windows XP+ / Server 2003+ (IE10+, Chrome Recent, Opera 15+)

1.2 Use jQuery 2.x

Kendo UI mobile widgets are support on windows phone 8+ and on IE 10+. Both use the ie10 browser rendering engine so you might as well use the newest version of jQuery, which drops support for IE 8 and below.

Notes
  • While Kendo should work with jQuery the official release is tested and shipped with 1.X jQuery.

1.3 Build common sense code

The following are common rules, laws, and practices among seasoned application developers for building long lasting applications through consistency:

  1. Entire code based should follow the same style of coding as if a single developer (i.e. mind) crafted the code. This should be validated automatically at code time (i.e. in IDE on key stroke), build time, and commit time (e.g. GIT Hooks).
  2. When writing code be verbose, priorities readability (i.e. code is read much more often than it is written), comment everything, and dispense with DRY principles if it obscures developer clarity.
  3. The DNA of an application is small miniature applications/components, properly decoupled into single responsibility modules.
  4. Modules are separated by nature (i.e. behavior/controller, model/data, style, template/html, test etc..) but organized together (i.e. application structure) by feature.
  5. Avoid complexity that does not pay for itself.
  6. Don't optimize development code, optimize production code (i.e. don't pre-optimize)
  7. Document everything!
  8. Automate what you can using a task runner to build code. Don't rely on human tasking.
  9. Establish at least one layer (unit, integration, functional) of automated testing that runs during build or before accepting code commits (i.e. Use a githook to run tests, reject if tests fail).
  10. Automate the logging and notification of runtime errors for production code.
  11. Fear monolithic abstractions in the form of closed frameworks which promote one stop developer solutions.

1.4 Develop using real devices

Simulators and browsers running on a desktop or laptop provide a quick and easy way to run Kendo UI Mobile code. However, this should not be how you test your code alone. It's critical that during development, development code is tested on the devices it intended to run on. I recommend at the very least testing on:

  • windows phone / windows mobile 8.0+
  • Android phone / Android 4.0+
  • iphone / iOS 6.0+

1.5 Make use of remote debugging tools

Plug a android (4.2+) or ios (iOS6) device into you computer/laptop and you can use the developer tools found in Safari or Chrome locally to debug the site running in the devices browser.

  • Android mobile debugging: Docs | Learn | Watch
  • iOS mobile debugging: Docs | Learn | Watch
Notes
  • The browser on a phone/tablet is representative of the engine (i.e. same engine) used in a webview and can be preemptively used to test hybrid applications.

1.8 Use normalize.css

When using the kendo.mobile.flat.min.css as a baseline theme with the intentions of customizing the theme further it can be beneficial to also include normalize.css.

The benefits of including normalize.css are:

  • Preserve useful browser defaults rather than erasing them.
  • Normalize styles for a wide range of HTML elements.
  • Correct bugs and common browser inconsistencies.
  • Improve usability with subtle improvements.
Notes
  • If you are using a CSS framework check to make sure that the CSS framework is not already using normalize.css.

1.9 You'll need a mobile first CSS framework

Kendo UI does not lock you into or prohibit the use of mobile first CSS frameworks to aid UI development. I've found that even the simplest of custom interfaces can benefit from the solutions found in a CSS framework (grids, form styles, form layouts etc...).

Some battle test mobile first CSS frameworks are:

  • bootstrap
  • Skeleton
  • purecss.io
  • cardinal
  • kraken

If introducing another third-party dependency into the code is not something you prefer doing then just realize you'll more than likely have to create a miniature custom CSS framework for aiding the development of custom interfaces.

Notes
  • I'm using bootstrap in the whats-er-name. If you use bootstrap, don't use the optional bootstrap theme. It contains some CSS that might cause performance issues.
  • You might consider using the icons provided by the CSS framework. For example, if you use boostrap you could sidestep the kendo UI icons and use glyphicons.com

1.10 Only use Kendo UI to register events

Events should be declared in a view using data attributes in the following ways: 

                                    data-click="viewModelEvent"

or

                                    data-bind="click:viewModelEvent"

or

                                    data-bind="events{click:viewModelEvent,click:viewModelEvent}"

Additionally, specific touch events that are not MVVM or widget specific can be setup generically using the Touch widget.

                                   <div data-role="view">
                                        <div data-role="touch" data-tap="onTap">
                                            Tap me
                                        </div>
                                    </div>

                                    <script>
                                        function onTap(e) {
                                            console.log(e.touch.target + " was tapped");
                                        }
                                    </script>

If you add events in any other manner except the ones describe above you'll have to write your own custom logic to deal with event delays and missed touches. Don't do that. Just let Kendo UI deal with it.

1.11 Improving tap predictability

Touch events are somewhat unpredictable and do not reliably fire all the time. To fix this today you can add the following code to your project to improve iOS and windows.

                                    var app = new kendo.mobile.Application(document.body, {
                                        init: function() {
                                            //don't do this for android, causes more issues than it fixes
                                           kendo.UserEvents.defaultThreshold(kendo.support.mobileOS.device === 'android' ? 0:20);
                                        }
                                    });

2. Kendo Mobile don'ts

2.1 Don't use view transitions

As of Kendo UI Q3 transitions are not worth the following issues that they either cause or exacerbate.

  • Complicates if not exacerbates the issue of drop touches/taps
  • Performance issues with animating views with complex content.
  • Transitions reliable work once and then on future uses fail and then exacerbate the events issues.

2.2 Don't turn on native scrolling

By default scrolling the content in a Kendo mobile app is not native. Either the scroll widget is used or CSS is used (i.e. overflow:auto) to emulate scrolling. While one can use the configuration below to force native scrolling, I won't suggest using it. All sorts of CSS issues and regression issues rear their ugly head if you turn on native scrolling.

                                new kendo.mobile.Application(document.body, { 
                                    useNativeScrolling: true 
                                });

2.3 Don't add mobile rendering meta elements

The meta elements below are automatically added to an html page when an application is instantiated.

                                <meta name="apple-mobile-web-app-capable" content="yes">
                                <meta name="apple-mobile-web-app-status-bar-style" content="black">
                                <meta name="msapplication-tap-highlight" content="no">
                                <meta content="initial-scale=1.0, maximum-scale=1.0, user-scalable=no" name="viewport">

Rely on this fact and don't add your own.

2.4 Don't use Internet Explorer compatibility modes

Internet Explorer compatibility modes can exhibit different behavior and rendering bugs, compared to the browser versions they emulate. It is highly recommended to use IE's Edge mode via META tag or HTTP header. The meta tag in question shown below.

                                <meta http-equiv="X-UA-Compatible" content="IE=edge" />

2.5 Don't use platform adaptive CSS

By default Kendo UI will use platform themes (e.g. first three devices below). However its better to force Kendo Mobile to use the kendo.mobile.flat.min.css for all platforms (e.g. far right device below) and completely avoid platform adaptive rendering.

To override Kendo UI's default nature of detecting the platform and forcing a platform specific CSS theme simply set the skin option to flat in the Application() method used to initialize the mobile application. 

                                var app = new kendo.mobile.Application($(document.body), { skin: 'flat' });

The reasons to do this is:

  1. Mimicking the look and feel of a specific platform turns out to be a complicated proposition resulting in a UI that will likely fall short of native look and feel, function, and performance in most cases. It's better to have a custom UI than to have a UI that is always in second place to the real thing.
  2. Using the platform neutral theme results in a fast user experience because the CSS used to create the neutral theme is void of the complex CSS (i.e. Gradients etc...) required by the platform themes to replicate native UI's.
  3. Overwriting and evolving one theme (i.e. the flat theme) is simpler than overwriting and evolving several platform specific themes. This is exactly what is done at http://kendoflatthemes.com/

2.6 Don't AJAX navigate to views (i.e. remote views)

Use a build step (like in the example application) to get all the views in the application into a single .html page so that the application does not have to AJAX in views when the application is running.

2.7 Don't fight against declarative widget initialization or MVVM

Kendo mobile widgets and application instance is integrated with the declarative widget initialization (i.e. create widgets from markup) and the MVVM pattern. Learn it, accept it, and don't fight it. Other wise you'd be better off not using the application and SPA logic found in the Application instance. You'd be better of either going all in on AngularJS or using the mobile widgets in stand alone mode.

2.8 Don't mix AngularJS with Kendo UI MVVM or Mobile SPA logic

If you are using AngularJS then you probably want to use Angular's own routing mechanism and data binding. Do not mix that with Kendo MVVM or Kendo mobile SPA logic (i.e. kendo.mobile.Application)

2.9 Don't use a dependency manager with mobile views

Assuming, and I do in this guide, you are using the mobile application logic don't attempt to load mobile views with a dependency manager. This is just unnecessary complexity overhead because in truth, because of the way that the application logic works all views have to be loaded and depended upon once (due to declarative instantiation) before the Application instance can be instantiated. This fact makes something like require.js almost pointless due to the complexity it adds to an application.

favor a build strategy (via a tasking runner e.g. Gulp or Grunt) that allows you to keep things in organized modules, but during the build process pulls everything into a single .html file like Kendo UI mobile assumes.

3. whats-er-name Folder Structure

3.1 Application structure overview

The image belows details the folder structured used to create the whats-er-name application.

You can investigate and navigate this directory structure yourself by getting the code from Github and running a dev build and production build using Gulp.

3.2 Bower & NPM directory setup

Bower and NPM are the package managers which make downloading third-party code (i.e. packages) trivial. I describe below how these are being used in the whats-er-name application.

Node packages (used by Gulp) are stored in a directory called node_modules at the root of directory containing the application. This is the default directory name and location, used by NPM, for storing node modules. The package.json file, which defines which modules are need, can also be found at the root of directory containing the application.

Bower, unlike npm, is not being used in the default manner. The bower.json file is in the root of the application directory, which is common, but I am using the .bowerrc file to configure bower to store packages in the wwwDev directory in a folder called bower_components.

3.3 Src, wwwDev, wwwProd directories

The whats-er-name application source code can be found in the src directory. This will be where you edit code during development. The wwwDev directory and the wwwProd directory will contain development and production versions of the code which get served to a web browser using gulp-webserver (more on that in the build and task section).

Notes
  • If you are looking at the code in Gitub the wwwDev and wwwProd folders are missing. This is because they are created from running the dev and production build tasks.

3.4 Shared directory

The src/shared directory will contain logically named and grouped modules which are either used globally to run the application or are used by more than one view module. Common files to find in the shared directory are global CSS modules, global layout modules, models, and utility modules. The shared essential to keeping things DRY. If more than one module in the UI directory uses another module likely the case you'll find in the shared directly in a logically named directory.

3.5 ui directory

The ui directory contains logically group and named UI portions of the application. It's not a hard and fast rule but think of the UI directory as the place you start dividing the application into a folder structure of features or logical sections of the application. Often the structure of this directory will mimic the architecture of the application from a navigation or tree map persecutive. This can often be a source of great confusion for developers because the structure of the folders depends greatly upon what you are building.

For a Kendo UI mobile application it might be best to simply consider that each view has its own directory potentially containing:

  • view-model JS file(aka a controller)
  • an HTML file containing the view
  • an HTML file containing a view specific layout
  • a CSS file containing view specific styles
  • a JS model file
  • any templates used by the view
  • a JS test file (Unit testing either the view-model, functionally testing the view)

These directories of views form a tree/hierarchical structure based on the logical break up of the application from the users perspective.

How exactly the ui directory is organized is not an exact science. What is important is that the directory is human friendly, development friendly, and its architecture facilitates an application made up of small parts, grouped logically together, that have a clear separation of concerns (i.e. small modules). Every folder in the UI directory should represent a logical part (i.e. a region of logically relevant ui) of the application. Inside of the folder will be all of the source code for that part only (excluding parts found in the shared directory that are globally used by more than one views).

If this is difficult for you to envision, study the whats-er-name wireframe and the whats-er-name directory structure (specifically the ui directory) to get a feel for this style of organization.

4. whats-er-name Builds & Application Servers

4.1 What is gulp doing

The application uses gulp (a task runner) to create development and production builds that are opened in a browser and severed from a local node server. Basically when developing, a development build is occurring (i.e. a bunch of tasks you don't want to run manually) every time you make a change to the source code. Note the browser reloads the page serving the development code (i.e. watching & livereload).

When you are done developing the application, a production build can be ran converting your development code into production. Note that production code is severed on a different server than development code and is not listening to changes from development directory for live reloading.

Gulp tasks are configure in the gulpfile.js file found in the root of the directory of the application. You'll find the file is broken up into wwwDev and wwwProd tasks.

4.1 The wwwDev build process & server

The tasks being done by gulp when running $ gulp dev from the command line are:

  1. empty the wwwDev directory except for bower components
  2. parse index.html file in src directory doing file includes, place all .html files (including update index.html) from src into wwwDev directory
  3. start a webserver at localhost:3027 serving the wwwDev directory, open localhost:3027 in default browser on local machine
  4. Listen for any change in src directory. If a change occurs run step 2
Notes
  • I'm not using any sort of CSS or JS pre-processors. But if you wanted too, you could easily adjust the wwwDev process to add them.

4.2 The wwwProd build process & server

The tasks being done by gulp when running $ gulp prod from the command line are:

  1. empty the wwwProd directory
  2. take wwwDev code, concatenate css and js based on build comments in index.html, version each file (e.g. whatsername-8b890cb0.js), place new code in wwwProd
  3. move all the image and font assets in wwwDev required by CSS to correct location in wwwProd
  4. lint CSS
  5. hint HTML
  6. hint JavaScript
  7. minify CSS, HTML, and JS
  8. start a web server at localhost:3028 serving the wwwProd directory, open localhost:3028 in default browser on local machine

5. whats-er-name Application Architecture

5.1 index.html overview

There will be only one valid HTML file (i.e. doctype, header, etc..) in the entire application. It will be called index.html and it will reside in the src folder. all other assets will be consider a module that is included via index.html using a <script> element, <link> element, or the build process (i.e. client side includes). Consider index.html the master configuration file for including all the necessary modules (models, views, view-models, layouts, templates, CSS, JS etc..) to make the application run. If you open the file and read the comments it should be obvious that running index.html bootstraps and instantiates the entire application.

5.2 Application name space

The Kendo UI mobile application by default looks for JavaScript objects in the global scope. The whats-er-name application uses a global name space to store JavaScript code relating to the application. The name space and shortcut to the name space are shown below and defined in the index.html file.

                                <script> this.whatsername = this.wrn = {}; //long and shortcut namespace </script>
Notes
  • Obviously more sophisticated solutions for dealing with scope and modules (i.e. dependency management/injection) are available today. But layering these abstractions over the top of the application logic adds unnecessary complexity. Using nested objects and having to type out longer object references is recommended because it works with the default behavior of Kendo, not against it. If you are using Angular, then you'd obviously be going down the Angular road (i.e. dependency injection) leaving behind all the Kendo mobile application logic.

5.3 Run/instantiation application

The mobile application is instantiated by the last script parsed in the index.html file. This is where you'd provide any settings for the kendo.mobile.Application instance. Below I am showing the JavaScript which instantiates the application.

                                    wrn.app = new kendo.mobile.Application($(document.body), { 
                                        skin: 'flat',
                                        initial: 'groupListView', //this is the initial view to load
                                        init: function() {
                                            //fix mouse events in iOS don't do it for android, causes more issues than it fixes
                                            kendo.UserEvents.defaultThreshold(kendo.support.mobileOS.device === 'android' ? 0:20);
                                        }
                                    });
