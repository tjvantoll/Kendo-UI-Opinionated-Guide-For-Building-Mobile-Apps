Introduction

What is this?

This is an opinionated guide for building websites, applications, and hybrid applications running on phones and tablets using Kendo UI mobile widgets and application tools. The intent is for the prescription(s) found within this document to become a style used by developers for building Kendo UI mobile sites/applications

Why do I need Kendo UI Mobile?

Kendo UI Mobile provides JavaScript widgets to be used in the construction of phone and tablet web based interfaces. In addition to widgets, Kendo UI Mobile also provides mobile centric application logic for building a Single Page Application in a browser or webview (i.e. Application, View, Layout). While Kendo UI Mobile widgets can run stand alone without an active mobile application instance this guide assumes the use of the mobile application logic in conjunction with the mobile widgets.

Who should read this

You are front-end engineer who has experience developing applications in HTML, CSS, and JavaScript (e.g. SPA applications) that wants to use these skills to build a mobile/tablet web site/application or a hybrid native application and are in need of some direction/opinions to get you started.

1. General Kendo UI Mobile Guidelines

1.1 Take stock of technologies available on the web platform

Kendo UI mobile, even if it's running in a webview, is render by a browser engine (i.e. a webview is like a browser window tab). Make sure to take note of the HTML5 (aka JavaScript API's), CSS, and JavaScript standards that are support by the platform you are targeting.

It's more than likely given the Kendo UI mobile support matrix (noted below) that your application will be running in a newer, modern browser engine, that supports things like ES5, CORS, and Web Sockets. Make sure you are aware of everything you have at your disposal for development and use it.

Notes
  • Kendo UI mobile is supported on the following operating systems and browser combinations Android 2.3+, iOS 6+, Windows Phone 8+, Chrome for Android, OS X 10.5+ (IE10+, Chrome Recent, Opera 15+, Safari 5+), and Windows XP + / Server 2003 + (IE10+, Chrome Recent, Opera 15+)

1.2 Use jQuery 2.x

Kendo UI mobile widgets are support windows phone 8+ and on IE 10+. Both use the ie10 rendering engine so you might as well use the newest version of jQuery, which drops support for IE 8 and below.

1.3 Don't use Internet Explorer compatibility modes

Internet Explorer compatibility modes can exhibit different behavior and rendering bugs, compared to the browser versions they emulate. It is highly recommended to use IE's Edge mode via META tag or HTTP header.

                                    <meta http-equiv="X-UA-Compatible" content="IE=edge" />

1.4 Favor the platform neutral flat theme for all platforms

Mimicking the look and feel of a specific platform turns out to be a complicated proposition. For this reason force Kendo Mobile to use the kendo.mobile.flat.min.css for all platforms and use either the mobile themebuilder to alter the theme or create your own overwrites in a separate style sheet.

To override Kendo UI's default nature of detecting the platform and forcing a platform specific CSS theme simply set the skin option to flat in the Application() method used to initialize the mobile application. 

                                    var app = new kendo.mobile.Application($(document.body), { skin: 'flat' });

1.5 Don't add mobile rendering meta elements

Kendo UI Mobile will add the appropriate rendering meta elements to the html page. Rely on this fact and don't add your own.

1.6 Build common sense code

The following are common rules, laws, and practices among seasoned application developers for building long lasting applications through consistency:

  1. Entire code based should follow the same style of coding as if a single developer (i.e. mind) crafted the code. This should be verified automatically at code time (i.e. in IDE on key stroke), build time, and commit time (e.g. GIT Hooks).
  2. When writing code be verbose, priorities readability (i.e. code is read much more often than it is written), comment everything, and dispense with DRY principles if it obscures developer clarity.
  3. The DNA of an application is small miniature applications/components, properly decoupled into single responsibility modules.
  4. Modules are separated by nature (i.e. behavior/controller, model/data, style, template/html, test etc..) but organized together (i.e. application structure) by feature.
  5. Avoid complexity that does not pay for itself.
  6. Don't optimize for humans during development, optimize staging and production code (i.e. don't pre-optimize)
  7. Document everything!
  8. Automate what you can through build processes and task runners.
  9. Establish at least one layer (unit, integration, functional) of automated testing.
  10. Automate the logging and notification of runtime errors minimally for production code.
  11. Fear monolithic abstractions in the form of closed frameworks which promote one stop developer minded solutions.

1.7 Test on a real device ASAP!

While simulators and browsers running on desktop/laptops provide a quick and easy way to test Kendo UI Mobile code testing should happen on a real device. Don't accept anything less than the real thing!
